# Интеграция с Vtiger CRM

## Как устроена авторизация в Vtiger

В Vtiger **два типа учёта**:

| | Вход в веб-интерфейс | REST API |
|---|----------------------|----------|
| **Логин** | username | username |
| **Секрет** | **пароль** (из таблицы пользователей) | **Access Key** (не пароль!) |
| **Где взять** | назначается при создании пользователя | Мой профиль → Настройки → Ещё → Сменить Access Key |

**Access Key** — отдельный токен только для API. Пароль из веб-логина в REST **не используется**.

---

## Варианты «API» под твой кейс

### 1. Использовать REST Vtiger как есть (рекомендуется)

- **Авторизация**: HTTP Basic Auth = `логин` + `Access Key`.
- **Базовый URL**: `https://ТВОЙ_VTIGER/restapi/v1/vtiger/default`
- **Документация**: [Rest API Manual](https://help.vtiger.com/article/147111249-Rest-API-Manual), [VTAP REST](https://vtap.vtiger.com/platform/rest-apis.html).

**Плюсы**: ничего не писать, всё из коробки.  
**Минус**: в расширении пользователь вводит **логин + Access Key** (не пароль).

В расширении добавлен **режим Vtiger**: в настройках включаешь «Режим Vtiger», в форме входа — «Логин» и «Access Key». Запросы к API идут с Basic Auth.

---

### 2. Свой тонкий прокси: логин + пароль из таблицы пользователей

Если нужен именно **логин и пароль** (как в веб-интерфейсе):

1. **Небольшой backend** (Node/PHP/Python на твоём сервере):
   - Таблица пользователей **или** прямая проверка через БД Vtiger (`vtiger_users` и т.п.).
   - Эндпоинт `POST /auth/login`: тело `{ "username", "password" }` → проверка пароля → выдаёшь **свой** JWT (или сессию).
   - Эндпоинты задач `GET/POST/PUT/DELETE /tasks` — только для авторизованных (JWT). Внутри прокси дергаешь REST Vtiger (Basic Auth: логин + **Access Key**).

2. **Откуда брать Access Key для вызовов Vtiger?**
   - **Вариант A**: при первом «Подключить Vtiger» пользователь один раз вводит логин + Access Key; сохраняешь их у себя (привязано к пользователю). Дальше в расширении он логинится только **логин + пароль** в твой прокси; прокси уже использует сохранённый Access Key для Vtiger.
   - **Вариант B**: один общий технический пользователь Vtiger с Access Key; все логины только в твой прокси. Пароли — твоя таблица или Vtiger, а в Vtiger всё летит от одного тех. пользователя.

3. Расширение тогда работает как сейчас: **Auth endpoint** = твой `https://.../auth/login`, **API URL** = твой `https://.../tasks`, Bearer токен от прокси.

**Итог**: «логин и пароль из таблицы юзеров» живут **в твоём прокси**. К самому Vtiger прокси ходит по Basic Auth (логин + Access Key).

---

### 3. Только проверка пароля по таблице Vtiger (без своего API)

- Делаешь сервис, который по `username` + `password` проверяет пользователя (например, по `vtiger_users`, с учётом хеша пароля Vtiger).
- Если ок — возвращаешь, условно, `{ "ok": true, "accessKey": "..." }`.  
- **Проблема**: Access Key в Vtiger хранится отдельно, его надо либо заранее знать, либо иметь к нему доступ (БД/внутренний API). Такой вариант возможен только если у тебя есть доступ к БД/внутренностям Vtiger и ты готов хранить маппинг user → access key.

На практике проще вариант 2 с прокси.

---

## Модули Vtiger под «задачи»

В Vtiger нет модуля «Tasks» в нашем понимании. Ближе всего:

- **Calendar** — события с датой/временем (аналог «задачи» с дедлайном).  
- **ProjectTask** — если пользуешься **Project** (проектами).

Для нашего расширения разумно маппить «задачи» на **Calendar** (или при необходимости на ProjectTask).

---

## Что сделано в расширении

В настройках **«Режим авторизации»**:

| Режим | Вход | Куда идут запросы |
|-------|------|-------------------|
| **Обычный API (Bearer)** | Логин + пароль → Auth endpoint → токен | API URL с Bearer |
| **Vtiger (Access Key)** | Логин + Access Key | API URL (Vtiger REST) с Basic Auth |
| **Vtiger (логин + пароль)** | Логин + пароль → **прокси** → accessKey | API URL (Vtiger REST) с Basic Auth |

- **API URL** — база Vtiger REST, например:  
  `https://ТВОЙ_VTIGER/restapi/v1/vtiger/default`
- **Vtiger (Access Key)**: Endpoint авторизации не нужен. Проверка через `GET {API_URL}/me` с Basic Auth.
- **Vtiger (логин + пароль)**: **Endpoint авторизации** = URL прокси (см. ниже). Расширение шлёт `POST { username, password }`, получает `{ accessKey, user }`, сохраняет и дальше ходит в Vtiger с Basic Auth (логин + accessKey).

---

## Прокси «логин + пароль» (vtiger_login_proxy)

В проекте есть папка **`vtiger_login_proxy`** с `login.php`. Прокси проверяет пароль по таблице `vtiger_users` и возвращает Access Key.

**Установка:**

1. Скопируйте папку `vtiger_login_proxy` на сервер с доступом к БД Vtiger (тот же сервер, что и CRM, или с сетевым доступом к MySQL).
2. **Вариант A** — свой config: `cp config.example.php config.php`, укажите `db_host`, `db_port`, `db_name`, `db_user`, `db_pass`.
3. **Вариант B** — конфиг Vtiger: положите прокси в **корень установки Vtiger** (рядом с `config.inc.php`). Будет использоваться та же БД.
4. В расширении: **Режим** = «Vtiger (логин + пароль)», **Endpoint авторизации** = `https://ваш-домен/vtiger_login_proxy/login.php`, **API URL** = база Vtiger REST.

**Контракт:**  
`POST` JSON `{ "username": "...", "password": "..." }` → 200 + `{ "accessKey": "...", "user": { "username": "..." } }` или 401 + `{ "message": "..." }`.

Дальше можно добавить маппинг «задачи» ↔ Calendar (или ProjectTask) и CRUD через `/query`, `/create`, `/update`, `/delete`.
